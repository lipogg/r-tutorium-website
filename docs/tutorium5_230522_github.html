2. Einstieg Git und GitHub

In der deutschsprachigen Wikipedia wird Git als eine “freie Software zur verteilten Versionsverwaltung von Dateien” definiert. Aber was bedeutet das konkret? Wenn ihr git auf eurem Rechner installiert habt, könnt ihr die Kommandozeile bzw. das Terminal oder wie in unserem Fall auch das RStudio dazu verwenden, um verschiedene Versionen einer Datei oder eines gesamten Directories (=Ordners) zu verwalten. Solche verwalteten Directories nennt man Repository.

Repositories können lokal oder auch online verwaltet werden, zum Beispiel über die Website github.com. Wenn ihr im Team zusammen an einem Projekt arbeiten wollt, lohnt es sich, das Projekt als GitHub Repository anzulegen. Alle Gruppenmitglieder können sich dann eine lokale Kopie des Projekts herunterladen und von ihrem eigenen Rechner aus Änderungen vornehmen und in das GitHub Repository hochladen. Daraus ergeben sich zwei Vorteile: Alle Teammitglieder sind immer auf dem aktuellen Stand und ihr könnt unkompliziert auf ältere Versionen des Projekts zugreifen, ohne, dass ihr für jede Version eine neue Datei anlegen müsst.

GitHub könntet ihr aber beispielsweise auch dazu nutzen, die (R-)Skripte, die ihr für ein Forschungsprojekt geschrieben habt, öffentlich zu teilen und Forschungsergebnisse zu präsentieren, zum Beispiel in Form einer über GitHub gehosteten Rmarkdown-Website wie dieser hier. Und nicht zuletzt könnt ihr über GitHub direkt an Open Source Softwareprojekten mitarbeiten, Probleme melden und Änderungsvorschläge machen und so in der Digital Humanities community aktiv werden.

In Aufgabe 2 habt ihr gleichzeitig ein neues Repository, ein Directory und ein RProject erstellt. Das sind viele neue Begriffe! Was ein Directory ist, wissen wir schon aus den letzten Stunden: Ein Directory ist ein lokales Verzeichnis auf eurem Computer, im Prinzip einfach ein Ordner mit ggf. Unterordnern, die dann Subdirectories heißen. Ein Repository ist letztlich nur ein verwaltetes Verzeichnis, das in unserem Fall der Versionsverwaltung dient. Und was ist ein RProject? Lest euch diesen Beitrag durch und beschreibt in euren eigenen Worten, was ein RProject ist.

3.4 Übung: Wiederholung und RStudio-GitHub Workflow

Über das RStudio kann man nicht nur ein neues Repository erstellen, sondern auch Repositories aus GitHub “klonen”, also als lokales verwaltetes Verzeichnis auf den eigenen Computer downloaden. Hier findet ihr eine Anleitung. Wenn man Zugangsrechte hat, kann man dann lokal Änderungen vornehmen und diese wieder auf GitHub “pushen”, also hochladen. Wenn mehrere Entwickler:innen an einem Projekt arbeiten, kann es so natürlich sein, dass jemand in unserer Abwesenheit eine Änderung vorgenommen hat, und somit die Version auf GitHub unserer lokalen Version eine Änderung voraus ist. Wenn wir lokal weiterarbeiten wollen, müssen wir diese Änderungen zuerst mit unserer lokalen Version synchronisieren. Dieser Vorgang wird “pull” genannt. Bevor man eine Änderung “pushen” kann, muss man sie “commiten”, also sozusagen zum pushen vormerken. Zu jedem Commit schreibt man eine kurze Nachricht, um sich später daran erinnern zu können, was für eine Änderung man vorgenommen hat. “Update README.md” heißt zum Beispiel, dass eine Änderung an der Datei “README.md” vorgenommen wurde.

Im RStudio sieht ein Git-GitHub-Workflow im Rahmen unseres Tutoriums wie folgt aus: Nachdem ihr das Repository zum Tutorium geklont habt, findet ihr in eurem Arbeitsverzeichnis einen Ordner r-tutorium. Öffnet die Datei r-tutorium.Rproj. Im RStudio seht ihr im Fenster unten rechts alle Dateien, die sich in eurem Verzeichnis r-tutorium befinden. Das Verzeichnis, in dem die RProject-Datei (.RProj) liegt, ist immer automatisch das Arbeitsverzeichnis für dieses Projekt. In unserem Fall ist das Arbeitsverzeichnis also das lokale Repository r-tutorium. Ihr könnt aus dem RStudio heraus jetzt zum Beispiel einen neuen Ordner anlegen (Fenster unten rechts “New Folder”), eine neue Datei anlegen (wie gehabt in der Menüleiste oben unter “File”) oder Änderungen an bestehenden Dateien im Repository vornehmen und die Änderungen anschließend “commiten” und in das “remote Repository” auf GitHub pushen. Zuerst solltet ihr aber eure lokale Version auf den neuesten Stand bringen und mögliche Änderungen aus dem GitHub-Repository pullen. Also:

  1. “Pull”: Fenster oben rechts -> Reiter “Git” -> blauer Pfeil, der nach unten zeigt

  2. Ordner oder Dateien anlegen oder Änderungen an bestehenden Dateien vornehmen

  3. “Commit”: Fenster oben rechts -> Reiter “Git” -> Dateien auswählen, die gepusht werden sollen -> Commit -> Commit-Nachricht eingeben -> Commit-Button

  4. “Push”: Fenster oben rechts -> Reiter "Git -> gründer Pfeil, der nach oben zeigt
